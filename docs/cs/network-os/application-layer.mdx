# Application Layer

## Why is it Important?

The application layer is **where backend engineers spend 80% of their time**. We work with HTTP, DNS, TLS, and gRPC every day, but many issues stem from insufficient understanding of protocol internal mechanisms.

### Real-World Impact

- **API Performance Bottlenecks**: HTTP/1.1 head-of-line blocking limits concurrency
- **DNS Resolution Failures**: Improper DNS cache configuration causes 50% request failures
- **TLS Handshake Failures**: Incomplete certificate chains prevent some clients from connecting
- **WebSocket Connection Drops**: Improper Keep-alive configuration causes connections to be dropped by intermediate devices

### After learning this section, you will be able to:

- Understand the differences and choices between HTTP/1.1, HTTP/2, and HTTP/3
- Configure and debug DNS, understand service discovery
- Understand TLS handshake flow, configure secure HTTPS
- Choose appropriate protocols (HTTP, WebSocket, gRPC)
- Optimize application layer performance and reduce latency

## HTTP Protocol

### HTTP/1.1 Basics

#### Request/Response Model

HTTP is a **stateless** request/response protocol:

```mermaid
sequenceDiagram
    participant Client as Client
    participant Server as Server

    Client->>Server: GET /users/1 HTTP/1.1
    Server-->>Client: HTTP/1.1 200 OK<br/>Content-Type: application/json

    Client->>Server: GET /users/2 HTTP/1.1
    Server-->>Client: HTTP/1.1 200 OK<br/>Content-Type: application/json

    Note over Client,Server: Each request needs separate response
```

#### Keep-Alive Connections

**Purpose:** Reuse TCP connections, reduce handshake overhead

**HTTP/1.0:** Connection closed by default

```http
GET /users/1 HTTP/1.0
Connection: close
```

**HTTP/1.1:** Connection kept alive by default

```http
GET /users/1 HTTP/1.1
Host: api.example.com
Connection: keep-alive  # Optional, enabled by default

GET /users/2 HTTP/1.1
Host: api.example.com
# Reuse same connection
```

**Performance Comparison:**

| Method | Total RTT for 3 requests | Latency (RTT=50ms) |
|------|----------------|------------------|
| **No Keep-Alive** | 6 RTT (2 RTT per request: connect+response) | 300ms |
| **Keep-Alive** | 3 RTT (connect once only) | 150ms |

**Configuration:**

```nginx
# Nginx
keepalive_requests 100;      # Max 100 requests per connection
keepalive_timeout 65s;       # Idle connection timeout: 65 seconds
```

#### Head-of-Line Blocking

**Problem:** HTTP/1.1 requests must be processed serially

```mermaid
sequenceDiagram
    participant Client as Client
    participant Server as Server

    Note over Client,Server: Connection established

    Client->>Server: GET /slow HTTP/1.1<br/>(takes 5 seconds)
    Note over Server: Processing...

    Client->>Server: GET /fast HTTP/1.1<br/>(takes 10ms)
    Note over Client: Waiting for /slow to complete

    Server-->>Client: 200 OK (/slow response)
    Server-->>Client: 200 OK (/fast response)

    Note over Client: /fast blocked by /slow for 5 seconds
```

**Solutions:**
- Multi-domain sharding
- HTTP/2 multiplexing
- HTTP/3 (QUIC)

### HTTP/2 Performance Revolution

**Released:** 2015 (based on Google SPDY)

**Core Improvements:**

#### 1. Binary Protocol

```
HTTP/1.1: GET / HTTP/1.1\r\nHost: example.com\r\n\r\n  (text)
HTTP/2:   0x00 0x01 0x02...                           (binary)
```

**Advantages:**
- More efficient parsing
- More compact

#### 2. Multiplexing

**Key Feature:** Send multiple requests concurrently over a single TCP connection

```mermaid
sequenceDiagram
    participant Client as Client
    participant Server as Server

    Note over Client,Server: Single TCP connection

    par Parallel requests
        Client->>Server: Stream 1: GET /slow
        Client->>Server: Stream 2: GET /fast
        Client->>Server: Stream 3: GET /data
    end

    par Parallel responses
        Server-->>Client: Stream 2: 200 OK (/fast)
        Server-->>Client: Stream 3: 200 OK (/data)
        Server-->>Client: Stream 1: 200 OK (/slow)
    end

    Note over Client,Server: Requests/responses parallel, no blocking
```

**Performance Impact:**

| Method | Total latency for 3 requests | Connections |
|------|----------------|--------|
| **HTTP/1.1** | 6 RTT | 1-3 |
| **HTTP/2** | 1 RTT | 1 |

**Backend Impact:**
- One connection per backend, not per request
- Simpler connection pool configuration

#### 3. Header Compression (HPACK)

**Problem:** HTTP headers transmitted repeatedly

```
Request 1: GET /users HTTP/1.1
        Host: api.example.com
        User-Agent: Mozilla/5.0...
        Accept: application/json

Request 2: GET /posts HTTP/1.1
        Host: api.example.com        # Duplicate
        User-Agent: Mozilla/5.0...    # Duplicate
        Accept: application/json     # Duplicate
```

**HTTP/2 Solution:** Static table + dynamic table + Huffman encoding

- Static table: Common headers (`:method`, `:path`, `:authority`)
- Dynamic table: Connection-level header cache
- Huffman encoding: Compress header values

**Performance Improvement:** Header size reduced by 80-90%

#### 4. Server Push

**Scenario:** Request HTML, server proactively pushes CSS/JS

```
Client: GET /index.html
Server: 200 OK (index.html)
        PUSH /style.css
        PUSH /app.js
```

**Status:** Disabled by most browsers (complex caching policies)

**Alternative:** Early Hints (103 status code)

### HTTP/3 and QUIC

**Released:** 2022

**Core Improvements:**

#### 1. Based on UDP

```
HTTP/1.1: TCP + TLS + HTTP/1.1
HTTP/2:  TCP + TLS + HTTP/2
HTTP/3:  QUIC (UDP) + HTTP/3
```

**Why Switch to UDP?**

| Issue | TCP (HTTP/2) | QUIC (HTTP/3) |
|------|-------------|---------------|
| **Head-of-Line Blocking** | Packet loss blocks all streams | Packet loss only affects single stream |
| **Connection Migration** | IP change requires reconnection | IP change seamless migration |
| **Handshake Overhead** | TCP (1 RTT) + TLS (1-2 RTT) | QUIC (0-1 RTT) |

#### 2. Solve TCP Head-of-Line Blocking

**HTTP/2 Problem:**

```
Stream 1: [Packet 1] [Packet 2 lost] [Packet 3]
Stream 2: [Packet 4] [Packet 5] [Packet 6]

TCP: Packet 2 lost Stream 1 and Stream 2 both blocked
```

**HTTP/3 Solution:**

```
Stream 1: [Packet 1] [Packet 2 lost] [Packet 3]
Stream 2: [Packet 4] [Packet 5] [Packet 6]

QUIC: Packet 2 lost Only Stream 1 blocked, Stream 2 continues
```

#### 3. Connection Migration

**Scenario:** Mobile device switches from WiFi to 4G

```
TCP (HTTP/2): IP change Connection disconnected Re-handshake
QUIC (HTTP/3): IP change Connection ID unchanged Seamless migration
```

**Backend Relevance:**
- Significant improvement for mobile users
- CDN traffic gradually migrating to HTTP/3

#### 4. 0-RTT Handshake

**Comparison:**

| Protocol | Handshake RTT | First request total latency |
|------|---------|----------------|
| **HTTP/1.1 + TLS 1.2** | 3 RTT | 150ms (RTT=50ms) |
| **HTTP/2 + TLS 1.3** | 2 RTT | 100ms |
| **HTTP/3 (0-RTT)** | 1 RTT | 50ms |

**0-RTT Principle:** Reuse previous session parameters

```
First connection:
  Client Server: ClientHello
  Server Client: ServerHello + Certificate
  Client Server: Finished
  (Complete)

Subsequent connections (0-RTT):
  Client Server: ClientHello + Application data
  Server Client: Finished + Response data
  (Complete)
```

### HTTP Methods and Semantics

#### Why HTTP Methods Matter

**Real-World Impact:**
- **API Design Quality**: Proper method usage creates intuitive, RESTful APIs
- **Caching Effectiveness**: Safe methods can be cached aggressively
- **Idempotency Guarantees**: Idempotent methods can be safely retried
- **Security**: Method semantics prevent accidental data modification

#### Method Properties

HTTP methods have three important properties that affect API design:

```mermaid
graph TD
    A[HTTP Methods] --> B[Safe]
    A --> C[Idempotent]
    A --> D[Cacheable]

    B --> B1[GET, HEAD, OPTIONS, TRACE]
    B --> B2[Does not modify server state]

    C --> C1[GET, HEAD, PUT, DELETE, OPTIONS, TRACE]
    C --> C2[Multiple identical requests = single request]

    D --> D1["GET, HEAD, POST (sometimes)"]
    D --> D2[Response can be stored and reused]

    style B fill:#e1f5e1
    style C fill:#fff4e1
    style D fill:#e1f5ff
```

**Property Definitions:**

| Property | Meaning | Methods |
|----------|---------|---------|
| **Safe** | Does not modify server state (read-only) | GET, HEAD, OPTIONS, TRACE |
| **Idempotent** | Multiple identical requests have same effect as one | GET, HEAD, PUT, DELETE, OPTIONS, TRACE |
| **Cacheable** | Response can be stored and reused | GET, HEAD, POST (condition) |

#### Method Deep Dive

##### 1. GET - Retrieve Data

**Purpose:** Request a representation of a resource

**Characteristics:** Safe, Idempotent, Cacheable

```http
GET /api/users/123 HTTP/1.1
Host: api.example.com
Accept: application/json

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 123,
  "name": "Alice",
  "email": "alice@example.com"
}
```

**Best Practices:**
- Use for read-only operations
- Keep request bodies empty (some servers reject body in GET)
- Use query parameters for filtering: `GET /users?status=active&page=2`
- Cache responses aggressively

**Common Mistakes:**
```bash
# ❌ DON'T: Use GET to modify data
GET /api/users/123/delete HTTP/1.1

# ✅ DO: Use DELETE for deletions
DELETE /api/users/123 HTTP/1.1
```

##### 2. POST - Create Resource

**Purpose:** Submit an entity to a resource (typically create new)

**Characteristics:** Not Safe, Not Idempotent, Conditionally Cacheable

```http
POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "name": "Bob",
  "email": "bob@example.com"
}

HTTP/1.1 201 Created
Location: /api/users/124
Content-Type: application/json

{
  "id": 124,
  "name": "Bob",
  "email": "bob@example.com"
}
```

**Best Practices:**
- Return `201 Created` with `Location` header for new resources
- Use for non-idempotent operations (create, trigger actions)
- Request body contains the resource representation
- Return `200 OK` or `204 No Content` for actions that don't create resources

**Use Cases:**
- Creating new resources
- Triggering server-side processes (send email, generate report)
- Querying with complex request bodies (GraphQL)

##### 3. PUT - Replace Resource

**Purpose:** Replace a resource with a complete new representation

**Characteristics:** Not Safe, Idempotent, Not Cacheable

```http
PUT /api/users/123 HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "id": 123,
  "name": "Alice Updated",
  "email": "alice.new@example.com"
}

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 123,
  "name": "Alice Updated",
  "email": "alice.new@example.com"
}
```

**Best Practices:**
- Send complete resource representation
- Idempotent: Multiple identical PUTs have same result
- Return `200 OK` (with updated resource) or `204 No Content`
- Can create resource if it doesn't exist (upsert)

**Common Mistakes:**
```bash
# ❌ DON'T: Send partial updates with PUT
PUT /api/users/123
{"email": "new@example.com"}  # Missing required fields!

# ✅ DO: Use PATCH for partial updates
PATCH /api/users/123
{"email": "new@example.com"}

# ✅ DO: Send complete resource with PUT
PUT /api/users/123
{"id": 123, "name": "Alice", "email": "new@example.com", ...}
```

##### 4. PATCH - Partial Update

**Purpose:** Apply partial modifications to a resource

**Characteristics:** Not Safe, Not Idempotent (depends on patch format), Not Cacheable

```http
PATCH /api/users/123 HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "email": "alice.updated@example.com"
}

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 123,
  "name": "Alice",
  "email": "alice.updated@example.com"
}
```

**Best Practices:**
- Send only fields to update
- Use for partial updates
- Idempotency depends on patch format:
  - JSON merge patch: Not idempotent (null values delete fields)
  - JSON patch: Idempotent if operations are idempotent

**PATCH Formats:**

| Format | Example | Idempotent |
|--------|---------|------------|
| **JSON Merge Patch** | `{"email": "new@example.com"}` | ❌ |
| **JSON Patch** | `[{"op": "replace", "path": "/email", "value": "new@example.com"}]` | ✅ |

```http
# JSON Patch (RFC 6902)
PATCH /api/users/123 HTTP/1.1
Content-Type: application/json-patch+json

[
  { "op": "replace", "path": "/email", "value": "new@example.com" },
  { "op": "remove", "path": "/phone" }
]
```

##### 5. DELETE - Remove Resource

**Purpose:** Delete a resource

**Characteristics:** Not Safe, Idempotent, Not Cacheable

```http
DELETE /api/users/123 HTTP/1.1
Host: api.example.com

HTTP/1.1 204 No Content
```

**Best Practices:**
- Return `204 No Content` (no response body)
- Idempotent: Deleting already-deleted resource = success
- Return `404 Not Found` if you want to distinguish existing vs non-existing
- Consider soft deletes for audit trails

**Response Options:**

| Status | When to Use |
|--------|-------------|
| **204 No Content** | Successful deletion (recommended) |
| **200 OK** | With confirmation message |
| **202 Accepted** | Async deletion (not yet complete) |
| **404 Not Found** | Resource doesn't exist |

##### 6. HEAD - Headers Only

**Purpose:** Get response headers without body

**Characteristics:** Safe, Idempotent, Cacheable

```http
HEAD /api/users/123 HTTP/1.1
Host: api.example.com

HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 156
Last-Modified: Wed, 18 Feb 2025 10:00:00 GMT
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"

# (No response body)
```

**Use Cases:**
- Check if resource exists and when it was last modified
- Check resource size before downloading
- Validate URL validity
- Test API endpoints (faster than GET)

**Example: Cache Validation**

```bash
# First request
curl -I https://api.example.com/users/123
# Returns: ETag: "abc123"

# Subsequent request with validation
curl -I -H "If-None-Match: \"abc123\"" https://api.example.com/users/123
# Returns: 304 Not Modified (no body, saves bandwidth)
```

##### 7. OPTIONS - Describe Options

**Purpose:** Describe communication options for target resource

**Characteristics:** Safe, Idempotent, Not Cacheable

```http
OPTIONS /api/users HTTP/1.1
Host: api.example.com

HTTP/1.1 204 No Content
Allow: GET, POST, PUT, DELETE, OPTIONS, HEAD
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS, HEAD
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400
```

**Use Cases:**
- CORS preflight requests (browser sends before actual request)
- Check allowed methods on a resource
- Check server capabilities

**Example: CORS Preflight**

```bash
# Browser sends preflight before cross-origin POST
OPTIONS /api/users HTTP/1.1
Origin: https://frontend.example.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type, Authorization

# Server responds with allowed methods
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://frontend.example.com
Access-Control-Allow-Methods: POST, GET, OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization
```

##### 8. TRACE - Echo Request

**Purpose:** Loop-back test for debugging (shows what server received)

**Characteristics:** Safe, Idempotent, Not Cacheable

```http
TRACE /api/users/123 HTTP/1.1
Host: api.example.com

HTTP/1.1 200 OK
Content-Type: message/http

TRACE /api/users/123 HTTP/1.1
Host: api.example.com
X-Forwarded-For: 10.0.0.1
```

**Use Cases:**
- Debugging proxy behavior
- Testing what headers intermediaries add
- **Security Risk:** Can expose sensitive information (cookies, auth headers)

**Security Warning:** Most production servers disable TRACE due to information disclosure risks.

#### REST API Design Best Practices

##### Resource Naming

```bash
# ✅ Good: Use nouns, plural form
GET    /users          # List users
GET    /users/123      # Get specific user
POST   /users          # Create user
PUT    /users/123      # Replace user
PATCH  /users/123      # Partial update
DELETE /users/123      # Delete user

# ❌ Bad: Use verbs in URL
GET /getUsers
POST /createUser
POST /deleteUser/123

# ❌ Bad: Use query parameters for identification
GET /users?id=123  # Use /users/123 instead
```

##### Nested Resources

```bash
# ✅ Good: One level nesting preferred
GET /users/123/orders        # Get orders for user 123
GET /users/123/orders/456    # Get specific order

# ⚠️ Acceptable: Multiple levels if needed
GET /users/123/orders/456/items/789

# ❌ Bad: Too deep nesting (hard to maintain)
GET /orgs/1/depts/2/teams/3/users/4/orders/5

# ✅ Better: Use query parameters for deep filtering
GET /orders?user=4&team=3&dept=2&org=1
```

##### Query Parameters for Filtering

```bash
# Filtering
GET /users?status=active
GET /users?role=admin&status=active

# Pagination
GET /users?page=2&limit=50
GET /users?offset=100&limit=50

# Sorting
GET /users?sort=created_at&order=desc

# Search
GET /users?q=alice

# Fields selection
GET /users?fields=id,name,email
```

#### Common Interview Questions

**Q1: Why is POST not idempotent while PUT is idempotent?**

A: POST creates a new resource on each request. Calling POST `/users` 10 times creates 10 different users. PUT replaces a resource at a specific URL. Calling PUT `/users/123` 10 times with the same data results in the same final state (user 123 has that data).

**Q2: When should I use PATCH vs PUT?**

A: Use PUT when replacing the entire resource. Send all required fields. Use PATCH when updating only specific fields. PATCH sends only changed fields, reducing bandwidth and avoiding partial update issues (forgetting required fields).

**Q3: What's the difference between 200 OK, 201 Created, and 204 No Content?**

A:
- `200 OK`: Request succeeded, response body contains data
- `201 Created`: Resource successfully created, include `Location` header
- `204 No Content`: Request succeeded, no response body (typical for DELETE, PUT)

**Q4: Why is GET called "safe" if it can have side effects (like logging, analytics)?**

A: "Safe" in HTTP spec means the method's primary purpose is not to modify server state. Safe methods can have side effects (logging, analytics) but these shouldn't be the primary purpose. Clients expect GET doesn't change resources, so they can safely retry, cache, and prefetch.

**Q5: How do I handle DELETE on an already-deleted resource?**

A: Since DELETE is idempotent, deleting an already-deleted resource should return success. Options:
- Return `204 No Content` (resource doesn't exist, which is the desired state)
- Return `404 Not Found` (if you want to distinguish existing vs non-existing)
- Return `200 OK` with confirmation message

**Q6: What's the purpose of the HEAD method, and when should I use it?**

A: HEAD returns headers without the body. Use cases:
- Check if resource exists (status code) and metadata (Content-Length, Last-Modified)
- Validate URL before downloading large files
- Check cache validity (ETag, Last-Modified) without transferring body
- Faster than GET for metadata-only requests

**Q7: How do I choose between POST and PUT for creating resources?**

A:
- **POST**: Server assigns resource ID. Client doesn't know URL beforehand.
  ```http
  POST /users
  # Server assigns ID 124, returns Location: /users/124
  ```

- **PUT**: Client assigns resource ID. Client knows the URL.
  ```http
  PUT /users/124
  # Client decides resource lives at /users/124 (upsert)
  ```

### HTTP Status Codes

#### Why Status Codes Matter

**Real-World Impact:**
- **API Usability**: Proper status codes make APIs intuitive and debuggable
- **Caching Strategy**: Status codes (200, 304) enable efficient caching
- **Error Handling**: Correct error codes guide client response behavior
- **Monitoring**: Status code distribution reveals system health

```mermaid
graph TD
    A[HTTP Status Codes] --> B[1xx: Informational]
    A --> C[2xx: Success]
    A --> D[3xx: Redirection]
    A --> E[4xx: Client Error]
    A --> F[5xx: Server Error]

    B --> B1[100 Continue<br/>102 Processing]
    C --> C1[200 OK<br/>201 Created<br/>204 No Content]
    D --> D1[301 Moved Permanently<br/>304 Not Modified<br/>307 Temporary Redirect]
    E --> E1[400 Bad Request<br/>401 Unauthorized<br/>404 Not Found<br/>429 Too Many Requests]
    F --> F1[500 Internal Server Error<br/>502 Bad Gateway<br/>503 Service Unavailable]

    style C fill:#e1f5e1
    style E fill:#fff4e1
    style F fill:#ffe1e1
```

#### Status Code Categories

| Category | Meaning | Common Codes |
|----------|---------|--------------|
| **1xx** | Informational | 100 Continue, 102 Processing |
| **2xx** | Success | 200 OK, 201 Created, 204 No Content |
| **3xx** | Redirection | 301 Moved Permanently, 302 Found, 304 Not Modified |
| **4xx** | Client Error | 400 Bad Request, 401 Unauthorized, 404 Not Found, 429 Too Many Requests |
| **5xx** | Server Error | 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable |

#### 2xx: Success Codes

##### 200 OK

**Meaning:** Request succeeded, response body contains requested data

**Use Cases:**
- GET request successful
- POST request processed (resource already existed or action completed)
- PUT/PATCH update successful

```http
GET /api/users/123 HTTP/1.1
Host: api.example.com

HTTP/1.1 200 OK
Content-Type: application/json
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"

{
  "id": 123,
  "name": "Alice",
  "email": "alice@example.com"
}
```

##### 201 Created

**Meaning:** Request succeeded and new resource created

**Required:** Include `Location` header with new resource URL

```http
POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "name": "Bob",
  "email": "bob@example.com"
}

HTTP/1.1 201 Created
Location: /api/users/124
Content-Type: application/json

{
  "id": 124,
  "name": "Bob",
  "email": "bob@example.com"
}
```

**Best Practice:** Return created resource in response body so client doesn't need to fetch it.

##### 202 Accepted

**Meaning:** Request accepted for processing, but processing not complete

**Use Cases:**
- Async operations (long-running tasks)
- Batch processing
- Background jobs

```http
POST /api/reports/generate HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "type": "monthly_sales",
  "format": "pdf"
}

HTTP/1.1 202 Accepted
Location: /api/jobs/abc123

{
  "job_id": "abc123",
  "status": "processing",
  "estimated_time": "5 minutes"
}

# Client polls for completion
GET /api/jobs/abc123

HTTP/1.1 200 OK
{
  "job_id": "abc123",
  "status": "completed",
  "result_url": "/api/reports/monthly_sales_2025_02.pdf"
}
```

##### 204 No Content

**Meaning:** Request succeeded, no response body

**Use Cases:**
- DELETE request successful
- PUT/PATCH update successful (no need to return updated resource)
- Action completed (toggle status, add to collection)

```http
DELETE /api/users/123 HTTP/1.1
Host: api.example.com

HTTP/1.1 204 No Content
```

**When to Use 204 vs 200:**
- Use `204` for DELETE, PUT, PATCH when client already sent the data
- Use `200` when returning updated representation or confirmation message

#### 3xx: Redirection Codes

##### 301 Moved Permanently

**Meaning:** Resource permanently moved to new URL

**SEO Impact:** Search engines update index to new URL

```http
GET /old-page HTTP/1.1
Host: example.com

HTTP/1.1 301 Moved Permanently
Location: /new-page
Content-Length: 0

# Browser automatically follows redirect
GET /new-page HTTP/1.1
Host: example.com

HTTP/1.1 200 OK
...
```

**Use Cases:**
- URL structure changed
- Domain migration
- Resource renamed

**Best Practice:** Keep 301 redirects in place indefinitely.

##### 302 Found (Temporary Redirect)

**Meaning:** Resource temporarily at different URL

**Behavior:** GET and POST requests use GET on redirect (historical bug)

```http
POST /submit-form HTTP/1.1
Host: example.com
Content-Type: application/x-www-form-urlencoded

name=Alice

HTTP/1.1 302 Found
Location: /success-page

# Browser changes POST to GET (loss of data!)
GET /success-page HTTP/1.1
```

**Warning:** Don't use 302 for POST requests. Use 307 instead.

##### 304 Not Modified

**Meaning:** Resource not modified (conditional GET, client cache valid)

**Use Cases:** Cache validation with ETag or Last-Modified

```http
# First request
GET /api/users/123 HTTP/1.1
Host: api.example.com

HTTP/1.1 200 OK
Content-Type: application/json
ETag: "abc123"
Last-Modified: Wed, 18 Feb 2025 10:00:00 GMT

{
  "id": 123,
  "name": "Alice"
}

# Subsequent request with validation
GET /api/users/123 HTTP/1.1
Host: api.example.com
If-None-Match: "abc123"

HTTP/1.1 304 Not Content
ETag: "abc123"
# (No body, saves bandwidth)
```

**Performance Impact:** 304 responses have no body, reducing bandwidth by 90%+ for cache hits.

##### 307 Temporary Redirect

**Meaning:** Temporary redirect, preserves method and body

**Use Cases:** Temporary URL changes, maintain POST data

```http
POST /checkout HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "item_id": 123,
  "quantity": 1
}

HTTP/1.1 307 Temporary Redirect
Location: /new-checkout

# Browser retries POST to new URL with same body
POST /new-checkout HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "item_id": 123,
  "quantity": 1
}

HTTP/1.1 201 Created
```

##### 308 Permanent Redirect

**Meaning:** Permanent redirect, preserves method and body

**Use Cases:** Permanent URL structure changes, maintain POST/PUT methods

#### Common Interview Questions

**Q1: What's the difference between 401 Unauthorized and 403 Forbidden?**

A: `401 Unauthorized` means authentication is required or failed (who are you?). `403 Forbidden` means the user is authenticated but lacks permission (I know you, but you can't do this). Example: Trying to access `/admin` without sending a JWT token = 401. Sending a valid token for a regular user trying to access `/admin` = 403.

**Q2: When should I return 200 OK vs 204 No Content?**

A: Use `200 OK` when returning data in the response body (GET success, POST/PUT/PATCH with updated resource). Use `204 No Content` when the request succeeded but there's no data to return (DELETE, PUT/PATCH when client already has the data). Example: DELETE `/users/123` → 204 No Content. GET `/users/123` → 200 OK with user data.

**Q3: What's the difference between 302, 307, and 301 redirects?**

A:
- `301 Moved Permanently`: Permanent redirect, SEO updates index, future requests go to new URL
- `302 Found`: Temporary redirect, historically changes POST to GET (bug, don't use for POST)
- `307 Temporary Redirect`: Temporary redirect, preserves HTTP method and body (use for POST)

For POST requests, always use 307 (not 302) to preserve the POST method and body.

**Q4: How does 304 Not Modified work with caching?**

A: Client sends conditional GET with `If-None-Match` (ETag) or `If-Modified-Since` (date). If resource unchanged, server returns 304 with no body. Client uses cached version. Saves bandwidth (no body transfer) and server processing (no data fetch). Example: First GET returns `ETag: "abc123"`. Subsequent GET includes `If-None-Match: "abc123"`. Server checks ETag, returns 304 if unchanged.

**Q5: When should I use 400 Bad Request vs 422 Unprocessable Entity?**

A: `400 Bad Request` for syntax errors (malformed JSON, missing required fields, wrong data types). `422 Unprocessable Entity` for semantic errors (business logic violations, complex constraints). Example: `{ "name": 123 }` where name must be string → 400 (wrong type). `{ "quantity": 1000 }` when only 500 in stock → 422 (business rule).

**Q6: What's the difference between 502 Bad Gateway and 504 Gateway Timeout?**

A: `502 Bad Gateway`: Upstream server returned invalid response (crashed, misconfigured, wrong protocol). `504 Gateway Timeout`: Upstream server didn't respond in time (slow processing, timeout). Both are gateway/proxy errors, but 502 means "got bad response" and 504 means "no response".

**Q7: How do I handle rate limiting in status codes?**

A: Return `429 Too Many Requests` with `Retry-After` header indicating seconds to wait. Include rate limit info in response body (limit, remaining, reset time). Client should wait `Retry-After` seconds before retrying. Example: `Retry-After: 60` means try again in 60 seconds.

### HTTP Caching Mechanisms

#### Why Caching Matters

**Real-World Impact:**
- **80% bandwidth reduction**: Cached responses eliminate redundant data transfers
- **50-90% latency reduction**: Serving from cache is 10-100x faster than origin
- **Server load reduction**: 70-95% fewer requests hit backend servers
- **CDN effectiveness**: Proper caching enables global content distribution

**Performance Example:**

| Scenario | Origin Load Time | Cached Load Time | Improvement |
|----------|-----------------|------------------|-------------|
| **No Caching** | 800ms (DB query + render) | 800ms | 0% |
| **Browser Cache** | 800ms | 50ms (disk read) | 94% |
| **CDN Cache** | 800ms | 20ms (edge hit) | 98% |

#### Cache-Control Directives

**Cache-Control** is the modern HTTP caching header (replaces older `Expires` header).

##### Response Directives

| Directive | Meaning | Use Case |
|-----------|---------|----------|
| **public** | Any cache can store (browser, CDN, proxy) | Static assets, public API responses |
| **private** | Only browser cache can store (no CDN) | User-specific data (authenticated requests) |
| **no-cache** | Must revalidate before using cached copy | API responses, dynamic data |
| **no-store** | Do not store anything (not even in memory) | Sensitive data (API keys, personal info) |
| **max-age=seconds** | Cache is fresh for N seconds | Static assets (images, CSS, JS) |
| **must-revalidate** | Must check freshness after expiration | Important data that shouldn't be stale |
| **immutable** | Resource never changes (cache forever) | Versioned assets (app.v1.js) |
| **stale-while-revalidate=seconds** | Use stale cache while revalidating in background | High-traffic endpoints |
| **s-maxage=seconds** | Override max-age for shared caches (CDN) | Longer CDN caching, shorter browser caching |

**Examples:**

```http
# Static assets (aggressive caching)
GET /static/css/main.css HTTP/1.1

HTTP/1.1 200 OK
Cache-Control: public, max-age=31536000, immutable
# Cache for 1 year, never changes

# API response (short cache + revalidation)
GET /api/users/123 HTTP/1.1

HTTP/1.1 200 OK
Cache-Control: public, max-age=60, must-revalidate
# Cache for 60 seconds, then revalidate

# User-specific data (browser only)
GET /api/me/profile HTTP/1.1
Authorization: Bearer token

HTTP/1.1 200 OK
Cache-Control: private, max-age=300
# Only browser caches (no CDN)

# Sensitive data (no caching)
GET /api/credit-cards HTTP/1.1
Authorization: Bearer token

HTTP/1.1 200 OK
Cache-Control: no-store, no-cache
# Never cache (credit card numbers)
```

##### Request Directives

| Directive | Meaning | Use Case |
|-----------|---------|----------|
| **no-cache** | Bypass cache, always fetch from origin | Force fresh data (user refresh) |
| **no-store** | Don't store response | Sensitive requests |
| **max-age=seconds** | Accept cached responses younger than N seconds | Override server defaults |
| **max-stale=seconds** | Accept stale responses up to N seconds | Offline mode, degraded experience |
| **min-fresh=seconds** | Want cache that stays fresh for N more seconds | Prefetching |

```bash
# Force bypass cache (get fresh data)
curl -H "Cache-Control: no-cache" https://api.example.com/data

# Accept stale data up to 1 hour (offline mode)
curl -H "Cache-Control: max-stale=3600" https://api.example.com/data
```

#### Entity Tags (ETag)

**ETag** is a validator (fingerprint) for resource versions.

**ETag Types:**

| Type | How Generated | Example | Change Detection |
|------|---------------|---------|------------------|
| **Strong ETag** | Content hash (MD5, SHA) | `"33a64df5..."` | Byte-level changes |
| **Weak ETag** | Last modified, entity size | `W/"33a64df5..."` | Semantic changes |

**Weak vs Strong:**

```http
# Strong ETag (changes if any byte changes)
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"

# Weak ETag (changes only for semantic modifications)
ETag: W/"33a64df551425fcc55e4d42a148795d9f25f89d4"
# Example: Adding whitespace doesn't change weak ETag
```

**Generating Strong ETag (Node.js example):**

```javascript
const crypto = require('crypto');

function generateETag(body) {
  return crypto.createHash('sha256')
    .update(body)
    .digest('base64');
}

// In Express
app.get('/api/users/:id', async (req, res) => {
  const user = await getUser(req.params.id);
  const body = JSON.stringify(user);
  const etag = generateETag(body);

  res.set('ETag', `"${etag}"`);
  res.send(body);
});
```

##### Conditional Requests (If-None-Match / If-Match)

**If-None-Match**: Cache validation (304 Not Modified if ETag matches)

```http
# First request
GET /api/users/123 HTTP/1.1
Host: api.example.com

HTTP/1.1 200 OK
Content-Type: application/json
ETag: "abc123"
Cache-Control: max-age=60

{
  "id": 123,
  "name": "Alice",
  "email": "alice@example.com"
}

# Subsequent request (after 60 seconds, cache expired)
GET /api/users/123 HTTP/1.1
Host: api.example.com
If-None-Match: "abc123"  # Client has cached version with this ETag

# Server checks if resource changed
HTTP/1.1 304 Not Modified  # Not changed, use cache
ETag: "abc123"
# (No body, saves bandwidth)
```

**If-Match**: Concurrency control (412 Precondition Failed if ETag doesn't match)

```http
# Client reads user (ETag: "abc123")
GET /api/users/123
# Returns ETag: "abc123"

# Client updates user (includes If-Match)
PATCH /api/users/123 HTTP/1.1
If-Match: "abc123"  # Only update if still version "abc123"
Content-Type: application/json

{
  "email": "new@example.com"
}

# Scenario 1: No concurrent modifications
HTTP/1.1 200 OK
ETag: "def456"  # New ETag after update
# Update succeeded

# Scenario 2: Another client already updated (ETag is now "xyz789")
HTTP/1.1 412 Precondition Failed
ETag: "xyz789"

{
  "error": "Resource was modified by another client",
  "current_etag": "xyz789"
}
# Update rejected, client should fetch latest version and retry
```

**Optimistic Locking with If-Match (Backend Implementation):**

```javascript
// Database version column
users: {
  id: 123,
  name: "Alice",
  email: "alice@example.com",
  version: 5,  // Increments on each update
}

// Generate ETag from version
function generateETag(user) {
  return `"user-v${user.version}"`;
}

// Update with concurrency check
app.patch('/api/users/:id', async (req, res) => {
  const user = await getUser(req.params.id);
  const ifMatch = req.get('If-Match');

  // Check ETag
  if (ifMatch && ifMatch !== generateETag(user)) {
    return res.status(412).json({
      error: 'Resource was modified by another client',
      current_etag: generateETag(user)
    });
  }

  // Update user (version auto-increments)
  await updateUser(user.id, req.body);
  user.version++;

  res.set('ETag', generateETag(user));
  res.json(user);
});
```

#### Last-Modified Validation

**Last-Modified**: Timestamp-based validator (older alternative to ETag)

```http
GET /api/users/123 HTTP/1.1

HTTP/1.1 200 OK
Last-Modified: Wed, 18 Feb 2025 10:00:00 GMT

# Subsequent request
GET /api/users/123 HTTP/1.1
If-Modified-Since: Wed, 18 Feb 2025 10:00:00 GMT

HTTP/1.1 304 Not Modified
# Resource not modified since that date
```

**If-Unmodified-Since**: Concurrency control (alternative to If-Match)

```http
PATCH /api/users/123 HTTP/1.1
If-Unmodified-Since: Wed, 18 Feb 2025 10:00:00 GMT
Content-Type: application/json

{
  "email": "new@example.com"
}

# If resource modified since that date
HTTP/1.1 412 Precondition Failed
```

#### ETag vs Last-Modified

| Feature | ETag | Last-Modified |
|---------|-------|---------------|
| **Accuracy** | Byte-level precision | Second-level precision |
| **Use Case** | Dynamic content | Static files |
| **Comparison** | Hash comparison | Timestamp comparison |
| **Performance** | Requires hashing | Already available |
| **Recommendation** | Use ETag (more accurate) | Use for static assets |

**Best Practice:** Use both ETag and Last-Modified

```http
HTTP/1.1 200 OK
ETag: "abc123"
Last-Modified: Wed, 18 Feb 2025 10:00:00 GMT
Cache-Control: max-age=60

# Client can use either for validation
GET /api/users/123
If-None-Match: "abc123"
If-Modified-Since: Wed, 18 Feb 2025 10:00:00 GMT
```

#### Vary Header and Content Negotiation

**Vary**: Specifies which request headers affect the response representation

**Use Case:** Different content for different Accept, Accept-Encoding, Accept-Language

```http
# Request 1: English content
GET /api/page HTTP/1.1
Accept-Language: en-US

HTTP/1.1 200 OK
Content-Language: en
Vary: Accept-Language
Cache-Control: max-age=3600

# Request 2: Spanish content
GET /api/page HTTP/1.1
Accept-Language: es-ES

HTTP/1.1 200 OK
Content-Language: es
Vary: Accept-Language
Cache-Control: max-age=3600
```

**Cache Key:** URL + Vary headers

```bash
# Cache stores separate entries:
# 1. GET /api/page + Accept-Language: en-US → English content
# 2. GET /api/page + Accept-Language: es-ES → Spanish content

# Wrong: Return English content for Spanish request (bug)
# Correct: Vary: Accept-Language ensures separate cache entries
```

**Common Vary Headers:**

```http
# Content negotiation
Vary: Accept, Accept-Language, Accept-Encoding

# Authenticated vs public content
Vary: Authorization, Cookie

# Mobile vs desktop
Vary: User-Agent

# Custom header
Vary: X-API-Version
```

**Warning:** Overusing Vary reduces cache hit rate (each header combination = separate cache entry).

#### Caching Strategies

##### 1. Static Assets (Aggressive Caching)

**Goal:** Never check origin again (until asset changes)

**Strategy:** Version URLs + long max-age + immutable

```http
# Versioned filename (cache busting)
GET /static/css/main.v1.2.3.css HTTP/1.1

HTTP/1.1 200 OK
Cache-Control: public, max-age=31536000, immutable
# Cache for 1 year, never revalidate

# When asset changes, deploy new version:
GET /static/css/main.v1.2.4.css HTTP/1.1
# New URL = cache miss
```

**Nginx Configuration:**

```nginx
location /static/ {
  # Cache static assets for 1 year
  expires 1y;
  add_header Cache-Control "public, immutable";

  # Enable gzip
  gzip on;
  gzip_types text/css application/javascript image/svg+xml;
}
```

##### 2. API Responses (Short Cache + Revalidation)

**Goal:** Reduce load while keeping data reasonably fresh

**Strategy:** Short max-age + must-revalidate

```http
GET /api/users/123 HTTP/1.1

HTTP/1.1 200 OK
ETag: "abc123"
Cache-Control: public, max-age=60, must-revalidate
# Cache for 60 seconds, then revalidate with ETag
```

**Timeline:**
```
0s:    GET request → cache miss → fetch from origin (ETag: abc123)
1s:    GET request → cache hit (fresh, 59s remaining)
60s:   GET request → cache expired → revalidate (If-None-Match: abc123)
       → 304 Not Modified (use cache, reset timer)
120s:  GET request → cache expired → revalidate
       → 200 OK (new data, ETag: def456)
```

##### 3. User-Specific Data (Browser Only)

**Goal:** Allow browser caching, prevent CDN caching (privacy)

**Strategy:** private + max-age

```http
GET /api/me/profile HTTP/1.1
Authorization: Bearer token

HTTP/1.1 200 OK
Cache-Control: private, max-age=300
# Only browser caches (no CDN or proxy)
```

**Why `private`?** CDN cache would serve user A's profile to user B (security issue).

##### 4. Sensitive Data (No Caching)

**Goal:** Never store sensitive information

**Strategy:** no-store + no-cache

```http
GET /api/credit-cards HTTP/1.1
Authorization: Bearer token

HTTP/1.1 200 OK
Cache-Control: no-store, no-cache
# Don't cache (not even in browser memory)
```

**Use `no-store` for:**
- Credit card numbers
- Personal identifiable information (PII)
- Authentication tokens
- Financial data
- Medical records

##### 5. High-Traffic Endpoints (Stale-While-Revalidate)

**Goal:** Serve stale cache immediately, revalidate in background

**Strategy:** stale-while-revalidate

```http
GET /api/products/popular HTTP/1.1

HTTP/1.1 200 OK
Cache-Control: public, max-age=60, stale-while-revalidate=300
# Fresh for 60s, then serve stale for up to 300s while revalidating
```

**Timeline:**
```
0s:    GET → cache miss → fetch from origin
60s:   GET → cache expired but stale-while-revalidate=300
       → serve stale immediately (fast response)
       → revalidate in background
       → update cache for next request
```

**CDN Configuration (Cloudflare example):**

```http
# Edge Cache TTL: 60s
# Serve Stale: 300s
Cache-Control: public, max-age=60, stale-while-revalidate=300
```

```mermaid
graph TD

    A -->|Yes, public| C{Who should cache?}
```

#### Common Caching Pitfalls

##### 1. Caching Authenticated Responses

**Problem:** CDN cache serves user A's data to user B

```http
# ❌ WRONG: Cache authenticated response
GET /api/me/orders HTTP/1.1
Authorization: Bearer user_a_token

HTTP/1.1 200 OK
Cache-Control: public, max-age=300  # CDN caches this!
# Now user B's request gets user A's orders
```

**Solution:** Use `private` for authenticated responses

```http
# ✅ CORRECT: Only browser caches
HTTP/1.1 200 OK
Cache-Control: private, max-age=300  # No CDN caching
```

##### 2. Forgetting to Cache Static Assets

**Problem:** Static assets (images, CSS, JS) returned with no caching headers

```http
# ❌ WRONG: No caching headers
GET /static/logo.png HTTP/1.1

HTTP/1.1 200 OK
# No Cache-Control header
# Browser re-downloads on every page load
```

**Solution:** Aggressive caching for static assets

```http
# ✅ CORRECT: Long cache
HTTP/1.1 200 OK
Cache-Control: public, max-age=31536000, immutable
```

##### 3. Overusing Vary Header

**Problem:** Too many Vary headers = low cache hit rate

```http
# ❌ WRONG: Too many Vary headers
GET /api/data HTTP/1.1
User-Agent: Mozilla/5.0...
Accept-Language: en-US
Accept-Encoding: gzip

HTTP/1.1 200 OK
Vary: User-Agent, Accept-Language, Accept-Encoding, Authorization, Cookie
# Each unique combination = separate cache entry
# Cache hit rate: &lt;1%
```

**Solution:** Only Vary necessary headers

```http
# ✅ CORRECT: Minimal Vary
HTTP/1.1 200 OK
Vary: Accept-Encoding  # Only vary on compression
```

##### 4. Cache Stampede

**Problem:** Cache expires, thousands of requests hit origin simultaneously

```
Time 0s:    Cache expires
Time 0.1s:  1000 requests arrive (cache miss, all hit origin)
Time 0.2s:  Origin overwhelmed, timeouts/crashes
```

**Solutions:**

1. **Stale-While-Revalidate:** Serve stale cache, revalidate in background
2. **Cache Lock:** Single request revalidates, others wait
3. **Randomized Expiry:** Add random jitter to cache expiry

```javascript
// Cache lock (Node.js example)
const cacheLocks = new Map();

app.get('/api/data', async (req, res) => {
  const key = req.url;

  // Check cache
  const cached = cache.get(key);
  if (cached && !isStale(cached)) {
    return res.send(cached.data);
  }

  // Check if another request is already updating
  if (cacheLocks.has(key)) {
    // Wait for existing request to finish
    await cacheLocks.get(key);
    return res.send(cache.get(key).data);
  }

  // Acquire lock
  const lock = Promise.resolve();
  cacheLocks.set(key, lock);

  // Fetch from origin (only this request)
  const data = await fetchFromOrigin(key);
  cache.set(key, data);

  // Release lock
  cacheLocks.delete(key);

  res.send(data);
});
```

#### Common Interview Questions

**Q1: What's the difference between no-cache and no-store?**

A: `no-cache` means the response can be stored but must be revalidated before use (conditional request with ETag/Last-Modified). `no-store` means don't store anything (not even in memory). Example: `no-cache` for API responses (validate freshness), `no-store` for credit card numbers (never cache).

**Q2: How does ETag-based cache validation work?**

A: First request: Server returns data + `ETag: "abc123"`. Client caches response with ETag. Subsequent request: Client sends `If-None-Match: "abc123"`. Server checks if resource changed. If unchanged: returns `304 Not Modified` (no body, saves bandwidth). If changed: returns `200 OK` with new data + new ETag.

**Q3: When should I use ETag vs Last-Modified?**

A: Use ETag for dynamic content (API responses, generated pages) because it's byte-level accurate. Use Last-Modified for static files (images, CSS) because it's simpler and already available. Best practice: Use both (ETag + Last-Modified) for maximum compatibility.

**Q4: What's the purpose of the Vary header?**

A: `Vary` tells caches which request headers affect the response representation. Example: `Vary: Accept-Language` means cache stores separate entries for English and Spanish content. Without Vary, cache might serve English content to Spanish users. Warning: Overusing Vary (many headers) reduces cache hit rate.

**Q5: How do I implement cache busting for static assets?**

A: Version the filename: `app.v1.2.3.js`. Deploy new version with new filename: `app.v1.2.4.js`. Old cache still serves `app.v1.2.3.js`, new requests fetch `app.v1.2.4.js`. Set long cache (`max-age=31536000, immutable`) because URL changes = cache busting.

**Q6: What's stale-while-revalidate and when should I use it?**

A: `stale-while-revalidate=N` allows serving stale cache for N seconds while revalidating in background. Use for high-traffic endpoints where fast response is more important than absolute freshness. Example: `Cache-Control: max-age=60, stale-while-revalidate=300`. After 60s, serve stale cache (fast) and revalidate (background). Reduces origin load during traffic spikes.

**Q7: How do I prevent cache stampede?**

A: Cache stampede = thousands of requests hitting origin when cache expires. Solutions: 1) `stale-while-revalidate`: serve stale, revalidate background. 2) Cache lock: single request revalidates, others wait. 3) Randomized expiry: add random jitter (±10%) to expiry time so requests don't expire simultaneously.

## Server-Sent Events (SSE)

### SSE Overview

**Server-Sent Events (SSE)** is a server-push technology enabling servers to stream data to clients over a single HTTP connection.

**Key Characteristics:**
- **One-way communication**: Server → Client (client cannot send messages to server)
- **Text-based**: Uses plain text format (UTF-8)
- **Automatic reconnection**: Browser automatically reconnects on disconnect
- **Event-based**: Supports named events and data types
- **Standard HTTP**: No special protocol (works through proxies/firewalls)

### SSE vs WebSocket Comparison

```mermaid
graph LR
    subgraph SSE
        A1[Server] -->|One-way| B1[Client]
        B1 -.->|Reconnects automatically| A1
    end

    subgraph WebSocket
        A2[Server] <-->|Bidirectional| B2[Client]
    end

    style B1 fill:#e1f5ff
    style A2 fill:#ffe1f5
    style B2 fill:#ffe1f5
```

| Feature | SSE | WebSocket |
|---------|-----|-----------|
| **Communication** | One-way (Server → Client) | Bidirectional (Full-duplex) |
| **Protocol** | HTTP (text/event-stream) | WebSocket protocol |
| **Browser Support** | 95% (except IE) | 96% |
| **Auto Reconnect** | ✅ Built-in | ❌ Manual implementation |
| **Proxies/Firewalls** | ✅ Works (standard HTTP) | ⚠️ May be blocked |
| **Binary Data** | ❌ Text only | ✅ Binary and text |
| **Use Cases** | Notifications, live updates | Chat, real-time collaboration |

**When to Use SSE:**
- Server needs to push updates to client
- Client doesn't need to send messages back
- Automatic reconnection is important
- Working through proxies/firewalls

**When to Use WebSocket:**
- Bidirectional real-time communication (chat, gaming)
- Binary data transfer
- Low latency is critical
- Client needs to send frequent messages

```javascript
    res.write(': heartbeat\n\n');
  }, 30000);

  // Handle client disconnect
  req.on('close', () => {
    clearInterval(interval);
    clearInterval(heartbeat);
    console.log('Client disconnected');
  });
});

const clients = new Set();

  res.setHeader('Cache-Control', 'no-cache');
  res.setHeader('Connection', 'keep-alive');
  res.flushHeaders();

  clients.add(res);

  req.on('close', () => {
    clients.delete(res);
  });
});

// Send broadcast to all clients
function broadcast(data) {
  const message = `data: ${JSON.stringify(data)}\n\n`;
  clients.forEach(client => {
    client.write(message);
  });
}
```

### Production Challenges

#### 1. Connection Limits

**Problem:** Each SSE connection keeps a server process/thread busy

**Scenario:** 10,000 users = 10,000 concurrent connections

- Increase server connection limits
- Use multiple servers with load balancer

#### 2. Reconnection ID Tracking

**Problem:** Client reconnects after disconnect, misses events during downtime

**Solution:** Server tracks last event ID, replays missed events

```javascript
const eventLog = [];  // In-memory event log

app.get('/api/events', (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');
  res.setHeader('Connection', 'keep-alive');
  res.flushHeaders();

  // Get Last-Event-ID from reconnecting client
  const lastEventId = parseInt(req.headers['last-event-id'] || '0');

  // Replay missed events
  const missedEvents = eventLog.filter(e => e.id > lastEventId);
  missedEvents.forEach(event => {
    res.write(`id: ${event.id}\ndata: ${JSON.stringify(event.data)}\n\n`);
  });

  // Send new events
  const interval = setInterval(() => {
    const event = {
      id: eventLog.length + 1,
      data: { message: 'Hello' }
    };
    eventLog.push(event);
    res.write(`id: ${event.id}\ndata: ${JSON.stringify(event.data)}\n\n`);
  }, 1000);
});
```

**Implementation (Node.js + Redis):**

```javascript
const subscriber = redis.createClient();

// Subscribe to Redis channel
subscriber.subscribe('events');

// Handle incoming Redis messages
subscriber.on('message', (channel, message) => {
  const event = JSON.parse(message);
  // Broadcast to all connected clients
  clients.forEach(client => {
    client.write(`data: ${JSON.stringify(event)}\n\n`);
  });
});

// SSE endpoint
app.get('/api/events', (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');
  res.setHeader('Connection', 'keep-alive');
  res.flushHeaders();

  clients.add(res);

  req.on('close', () => {
    clients.delete(res);
  });
});

// Publish event to Redis (any server)
app.post('/api/events/publish', (req, res) => {
  const event = req.body;
  publisher.publish('events', JSON.stringify(event));
  res.json({ success: true });
```

#### 4. Detecting Disconnected Clients

**Problem:** Server doesn't know client disconnected (keeps sending events to dead connection)

**Solutions:**

1. **Heartbeat (ping/pong):**

```javascript
app.get('/api/events', (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.flushHeaders();

  // Send heartbeat every 30s
  const heartbeat = setInterval(() => {
    res.write(': heartbeat\n\n');
  }, 30000);

  // Detect disconnect on socket error
  res.socket.on('error', () => {
    clearInterval(heartbeat);
  });

  req.on('close', () => {
    clearInterval(heartbeat);
  });
});
```

2. **Client echo (client replies to server messages):**

```javascript
// Server
app.get('/api/events', (req, res) => {
  let lastEcho = Date.now();

  // Check for echo (client responds within 10s)
  const echoCheck = setInterval(() => {
    if (Date.now() - lastEcho > 10000) {
      // Client not responding, close connection
      res.end();
      clearInterval(echoCheck);
    }
  }, 5000);

  // Listen for client echo (via separate endpoint)
  app.get('/api/events/echo', (req, res) => {
    lastEcho = Date.now();
    res.json({ pong: true });
  });
});

// Client
const eventSource = new EventSource('/api/events');

setInterval(() => {
  fetch('/api/events/echo');  // Send echo every 5s
}, 5000);
```

### Common Interview Questions

**Q1: When should I use SSE vs WebSocket?**

A: Use SSE for server→client streaming (notifications, live updates, progress tracking). Use WebSocket for bidirectional communication (chat, real-time collaboration). SSE advantages: auto-reconnect, works through proxies, simpler. WebSocket advantages: bidirectional, binary data, lower latency.

**Q2: How does SSE handle reconnection?**

A: Browser automatically reconnects after disconnect (default 3 seconds). Server includes `id` field in events. Client sends `Last-Event-ID` header on reconnect. Server replays missed events since last ID. Example: Server sends `id: 123`. Client disconnects, reconnects with `Last-Event-ID: 123`. Server sends events from id=124 onwards.

**Q3: How do I scale SSE across multiple servers?**

A: Use Redis Pub/Sub for cross-server communication. Each server has its own SSE clients. One server publishes event to Redis. All servers subscribe to Redis, receive event, broadcast to their clients. This ensures all clients receive events regardless of which server they're connected to.

**Q4: What happens if a client disconnects unexpectedly?**

A: Server continues writing to socket (doesn't know client disconnected). Detect disconnect via: 1) Socket error event, 2) Client heartbeat (client sends periodic ping), 3) Write error (next write fails). Clean up resources (close intervals, remove from client list). Missed events: If using event IDs, client reconnects with Last-Event-ID, server replays missed events.

**Q5: How do I handle SSE behind Nginx?**

A: Configure Nginx to disable buffering and keep connections alive:
```nginx
proxy_buffering off;           # Don't buffer SSE
proxy_cache off;               # Don't cache SSE
proxy_set_header Connection ''; # Enable keep-alive
proxy_read_timeout 3600s;      # Long timeout for SSE
```

**Q6: Can SSE send binary data?**

A: No, SSE only supports UTF-8 text. Workarounds: 1) Base64 encode binary data, 2) Send binary as JSON array of bytes, 3) Use WebSocket for binary data. Example: `data: {"binary": "SGVsbG8gV29ybGQ="}` (Base64 encoded).

**Q7: How do I prevent Nginx from buffering SSE responses?**

A: Add `X-Accel-Buffering: no` header to disable Nginx buffering. Configure Nginx with `proxy_buffering off;`. Send immediate heartbeat (comment) to flush headers: `res.write(': heartbeat\n\n')`.

### Backend Challenges

#### 1. Connection Management

**Problem:** Many connections consume memory and file descriptors

**Solution:**

```go
// Connection pool management
type ConnectionPool struct {
    connections map[*Conn]bool
    register    chan *Conn
    unregister  chan *Conn
    broadcast   chan []byte
}

func (p *ConnectionPool) run() {
    for {
        select {
        case conn := <-p.register:
            p.connections[conn] = true
        case conn := <-p.unregister:
            delete(p.connections, conn)
        case message := <-p.broadcast:
            for conn := range p.connections {
                conn.send <- message
            }
        }
    }
}
```

**Solution:**

### Email Security

#### SPF (Sender Policy Framework)

**Purpose:** Prevent sender spoofing by listing authorized mail servers

**DNS Record:**

```dns
example.com. IN TXT "v=spf1 ip4:192.0.2.0/24 include:_spf.google.com ~all"
```

**Explanation:**
- `v=spf1`: SPF version 1
- `ip4:192.0.2.0/24`: Authorize IP range
- `include:_spf.google.com`: Include Google's SPF record
- `~all`: Soft fail (accept but mark as spam if no match)
- `-all`: Hard fail (reject if no match, stricter)

**Verification Flow:**

```mermaid
    D -->|No SPF record| G[No SPF<br/>Neutral result]
```

#### DMARC (Domain-based Message Authentication, Reporting, and Conformance)

**Purpose:** Policy for handling emails that fail SPF or DKIM

**DNS Record:**

```dns
_dmarc.example.com. IN TXT "v=DMARC1; p=quarantine; rua=mailto:dmarc@example.com"
```

**Tags:**

| Tag | Meaning | Values |
|-----|---------|--------|
| **v=DMARC1** | DMARC version | DMARC1 |
| **p** | Policy (what to do on fail) | `none`, `quarantine`, `reject` |
| **sp** | Subdomain policy | `none`, `quarantine`, `reject` |
| **pct** | Percentage of emails to apply policy | 0-100 |
| **rua** | Send aggregate reports | `mailto:dmarc@example.com` |
| **ruf** | Send forensic reports | `mailto:forensic@example.com` |

**Policy Levels:**

| Policy | Action | Use Case |
|--------|--------|----------|
| **p=none** | Monitor only, don't enforce | Initial deployment, testing |
| **p=quarantine** | Move to spam folder | Confident in SPF/DKIM setup |
| **p=reject** | Reject email | Strict security, fully validated |

**DMARC Decision Flow:**


```java
        message.setText(text);

        mailSender.send(message);
    }

    // HTML email
    public void sendHtmlEmail(String to, String subject, String htmlContent)
            throws MessagingException {
        MimeMessage message = mailSender.createMimeMessage();
        MimeMessageHelper helper = new MimeMessageHelper(message, true, "UTF-8");

        helper.setFrom("alice@example.com");
        helper.setTo(to);
        helper.setSubject(subject);
        helper.setText(htmlContent, true);  // true = HTML

        mailSender.send(message);
    }

    // Email with attachment
    public void sendEmailWithAttachment(String to, String subject,
            String text, String attachmentPath) throws MessagingException {
        MimeMessage message = mailSender.createMimeMessage();
        MimeMessageHelper helper = new MimeMessageHelper(message, true);

        helper.setFrom("alice@example.com");
        helper.setTo(to);
        helper.setSubject(subject);
        helper.setText(text);

        // Add attachment
        FileSystemResource file = new FileSystemResource(attachmentPath);
        helper.addAttachment("invoice.pdf", file);

        mailSender.send(message);
    }
}
```

### Common Interview Questions

**Q1: What's the difference between SPF, DKIM, and DMARC?**

A: SPF verifies that the sending server is authorized (IP-based). DKIM verifies that the email content wasn't tampered (cryptographic signature). DMARC combines SPF and DKIM to define policy for failed emails (reject, quarantine, none). SPF = "Is this server allowed?". DKIM = "Is this email authentic?". DMARC = "What should we do with failures?".

**Q2: Why is port 25 often blocked by ISPs?**

A: Port 25 was designed for server-to-server SMTP, not client submission. ISPs block port 25 to prevent spam bots from sending emails directly. Use port 587 (STARTTLS) or 465 (SSL) for client submission, which require authentication.

**Q3: How do I prevent my emails from going to spam?**

A: 1) Set up SPF, DKIM, DMARC records. 2) Use consistent sending domain and IP. 3) Monitor sender reputation (check blacklist). 4) Avoid spammy content (excessive caps, spam words). 5) Include unsubscribe link (CAN-SPAM compliance). 6) Use dedicated IP for high volume. 7) Warm up new IP (start low, gradually increase).

**Q4: What's the SMTP flow for sending an email?**

A: 1) MUA connects to MTA (port 587). 2) EHLO/HELO handshake. 3) MAIL FROM (sender). 4) RCPT TO (recipient). 5) DATA (email content). 6) MTA relays to recipient MTA (DNS MX lookup). 7) Recipient MTA delivers to MDA. 8) MDA stores in mailbox. 9) Recipient MUA fetches via IMAP/POP3.

**Q5: How does email spoofing work and how do I prevent it?**

A: Spoofing = sending email with forged From address. Attackers set MAIL FROM to victim's email. Prevention: SPF (list authorized servers), DKIM (sign outgoing emails), DMARC (reject unsigned emails). Without SPF/DKIM, anyone can send email as "ceo@yourcompany.com".

**Q6: What's the difference between SMTP ports 25, 587, and 465?**

A: Port 25: Server-to-server SMTP (historical), often blocked by ISPs. Port 587: Client submission with STARTTLS (recommended), requires authentication. Port 465: Implicit SSL (deprecated but still used), SSL from connection start. Use port 587 with STARTTLS for modern email sending.

**Q7: How do I implement DMARC rollout safely?**

A: Start with `p=none, pct=100` (monitor only, no enforcement). Analyze DMARC reports (check legitimate email sources). After 1-2 weeks, move to `p=quarantine, pct=10` (quarantine 10% of failures). Gradually increase to `pct=50`, then `pct=100`. Finally, move to `p=reject, pct=100` (reject failures). Rollout takes 4-8 weeks.

### AMQP (Advanced Message Queuing Protocol)

**Purpose:** Reliable message queuing for enterprise systems

**Characteristics:**
- Reliable (acknowledgments, transactions)
- Flexible routing (exchanges, bindings)
- Interoperable (language-agnostic)
- Complex but powerful

**AMQP Architecture:**

```mermaid
graph LR
    A[Producer] --> B[Exchange]
    B -->|binding key| C[Queue 1]
    B -->|binding key| D[Queue 2]
    B -->|binding key| E[Queue 3]

    C --> F[Consumer 1]
    D --> G[Consumer 2]
    E --> H[Consumer 3]

    style B fill:#ffe1e1
    style C fill:#e1f5ff
    style D fill:#e1f5ff
    style E fill:#e1f5ff
```

**Exchange Types:**

| Type | Behavior | Use Case |
|------|----------|----------|
| **Direct** | Route to queue by exact routing key | Point-to-point messaging |
| **Fanout** | Broadcast to all queues | Pub/sub, notifications |
| **Topic** | Route by wildcard pattern | Filtered messaging |
| **Headers** | Route by header values | Complex routing rules |

**Node.js AMQP (RabbitMQ) Example:**

```javascript
const amqp = require('amqplib');

// Publisher
async function publishMessage() {
  const connection = await amqp.connect('amqp://localhost');
  const channel = await connection.createChannel();

  const exchange = 'logs';
  const routingKey = 'info';
  const message = 'Hello, AMQP!';

  await channel.assertExchange(exchange, 'topic', { durable: false });
  channel.publish(exchange, routingKey, Buffer.from(message));

  console.log(` [x] Sent ${routingKey}:${message}`);
  await connection.close();
}

// Subscriber
async function consumeMessages() {
  const connection = await amqp.connect('amqp://localhost');
  const channel = await connection.createChannel();

  const exchange = 'logs';
  const queue = await channel.assertQueue('', { exclusive: true });
  const bindingKey = '*.info';

  await channel.assertExchange(exchange, 'topic', { durable: false });
  channel.bindQueue(queue.queue, exchange, bindingKey);

  channel.consume(queue.queue, (msg) => {
    console.log(` [x] ${msg.fields.routingKey}:${msg.content.toString()}`);
  }, { noAck: true });
}
```

### Protocol Comparison

| Protocol | Type | Use Case | Pros | Cons |
|----------|------|----------|------|------|
| **HTTP** | Request/Response | Web services, APIs | Universal, simple | Latency, overhead |
| **WebSocket** | Bidirectional | Real-time chat | Low latency, bidirectional | Stateful, complex scaling |
| **SSE** | Server push | Notifications, live updates | Simple, auto-reconnect | One-way only |
| **SMTP** | Email | Email delivery | Universal, reliable | Spam, security issues |
| **FTP/SFTP** | File transfer | Bulk file transfer | Simple, resumable | Slow for many small files |
| **SSH** | Remote shell | Server administration | Secure, versatile | Not for data transfer |
| **MQTT** | Pub/Sub | IoT, sensors | Lightweight, low-power | Limited features |
| **AMQP** | Message queue | Enterprise messaging | Reliable, flexible | Complex, heavy |

### Common Interview Questions

**Q1: When should I use MQTT vs AMQP?**

A: MQTT for IoT, low-power devices, unstable networks (lightweight, simple). AMQP for enterprise messaging, reliable queues, complex routing (heavy, feature-rich). Example: Smart home sensors → MQTT. Banking transactions → AMQP.

**Q2: What's the difference between direct, fanout, and topic exchanges in AMQP?**

A: Direct exchange routes to queue by exact routing key match (point-to-point). Fanout exchange broadcasts to all bound queues (pub/sub). Topic exchange routes by wildcard pattern (filtered messaging, e.g., `logs.*` matches `logs.error`, `logs.info`).

**Q3: Why is SFTP preferred over FTP?**

A: SFTP encrypts everything (commands, data, passwords). FTP transmits passwords in plain text (security risk). SFTP uses standard SSH port (22) = better firewall compatibility. SFTP has better permission handling and file attributes.

**Q4: How does MQTT QoS work?**

A: QoS 0 (at most once): Fire and forget, no guarantee (fastest). QoS 1 (at least once): Acknowledged delivery, could duplicate (balanced). QoS 2 (exactly once): 4-way handshake, guaranteed exactly once (slowest). Example: Temperature sensor → QoS 0 (non-critical). Bank transaction → QoS 2 (critical).

**Q5: What's SSH tunneling and when should I use it?**

A: SSH tunneling forwards traffic through SSH connection (encrypted). Use cases: 1) Local forwarding: Access remote service via local port (`ssh -L 8080:remote:80`). 2) Remote forwarding: Expose local service to remote (`ssh -R 8080:localhost:3000`). 3) Dynamic (SOCKS) proxy: Browse via SSH tunnel (`ssh -D 1080`).

**Q6: How do I choose between MQTT, SSE, and WebSocket for real-time updates?**

A: MQTT for IoT, low-bandwidth, device-to-server. SSE for server-to-client updates (notifications, live feed). WebSocket for bidirectional real-time (chat, collaboration). Example: Sensor dashboard → MQTT (device) + SSE (browser). Chat app → WebSocket.

**Q7: What's the difference between message queues (AMQP) and pub/sub (MQTT)?**

A: Message queue: Each message consumed by one consumer (work distribution). Pub/sub: Each message broadcast to all subscribers (notifications). Example: Task processing → AMQP queue (1 worker per task). News updates → MQTT pub/sub (all subscribers receive).

### 8. SMTP Relaying Denied

**Symptom:** `550 5.7.1 Relaying denied` error

**Causes:**
- Not authenticated
- Wrong SMTP port (25 instead of 587)
- Sending from unauthorized IP

**Debugging:**

```bash
# Test SMTP connection
telnet smtp.example.com 587

# Test with authentication (swaks)
swaks --to recipient@example.com --from sender@example.com \
  --server smtp.example.com --auth-user sender@example.com \
  --auth-password password --tls

# Check if authentication works
openssl s_client -connect smtp.example.com:587 -starttls smtp
# Then: AUTH PLAIN <base64(username\0username\0password)>
```

**Solution:**

```javascript
// Use correct port (587) and authenticate
const transporter = nodemailer.createTransport({
  host: 'smtp.example.com',
  port: 587,  // Not 25
  secure: false,  // STARTTLS
  auth: {
    user: 'sender@example.com',
    pass: 'password'
  }
});
```

### Scenario 5: Notification System Architecture

**Background:** Need real-time notifications for web and mobile apps (1M+ users)

**Protocol Comparison:**

| Protocol | Pros | Cons | Verdict |
|----------|------|------|---------|
| **Polling** | Simple, universal | High server load, delayed updates | ❌ Not scalable |
| **SSE** | Simple, auto-reconnect | One-way only | ✅ Best for server→client |
| **WebSocket** | Bidirectional, low latency | Complex, stateful | ⚠️ Overkill for notifications |
| **MQTT** | Lightweight, IoT-friendly | Different protocol | ⚠️ Niche use cases |

**Solution:** SSE for web notifications + FCM (Firebase Cloud Messaging) for mobile

```mermaid
graph TD
    A[Event Occurs] --> B[Event Bus<br/>Kafka/RabbitMQ]
    B --> C[Notification Service]

    C --> D{Target Platform?}
    D -->|Web Browser| E[SSE Endpoint]
    D -->|Mobile App| F[FCM/APNS Push]

    E --> G[Browser receives<br/>real-time update]
    F --> H[Mobile receives<br/>push notification]

    C --> I[Database<br/>Notification History]
    G -.->|Poll on reconnect| I
    H -.->|App open| I

    style E fill:#e1f5ff
    style F fill:#ffe1e1
    style I fill:#fff4e1
```

**SSE Implementation:**

```javascript
// SSE endpoint with Redis Pub/Sub (multi-server)
app.get('/api/notifications', (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');
  res.flushHeaders();

  // Send missed notifications (from database)
  const lastEventId = parseInt(req.headers['last-event-id'] || '0');
  const missedNotifications = await getNotificationsSince(lastEventId);
  missedNotifications.forEach(n => {
    res.write(`id: ${n.id}\ndata: ${JSON.stringify(n.data)}\n\n`);
  });

  // Subscribe to Redis for new notifications
  const subscriber = redis.duplicate();
  await subscriber.subscribe('notifications');

  subscriber.on('message', (channel, message) => {
    res.write(`data: ${message}\n\n`);
  });

  req.on('close', () => {
    subscriber.disconnect();
  });
});

// Heartbeat every 30s (keep connection alive)
setInterval(() => {
  res.write(': heartbeat\n\n');
}, 30000);
```

**Scaling Strategy:**
- Multiple servers: Redis Pub/Sub broadcasts to all servers
- Reconnection: Client sends Last-Event-ID, server replays missed events
- Offline support: Store notifications in database, sync on reconnect

**Results:**
- 1M+ concurrent SSE connections
- Sub-second delivery latency
- 99.9% availability
