# Physical Layer

## Why is it Important?

The physical layer is the **foundation of the TCP/IP protocol stack**, responsible for transmitting bit streams over physical media. Although backend engineers rarely operate the physical layer directly, understanding it helps us:

- **Select Data Centers**: Understand why us-east-1 is faster for US East users
- **Understand Latency**: Identify different types of latency: propagation, serialization, etc.
- **Network Planning**: Understand trade-offs between single-region and multi-region deployment
- **Fault Location**: Determine if issues are at the application layer or physical layer (fiber cut)

### After learning this section, you will be able to:

- Understand characteristics of different transmission media
- Distinguish between bandwidth and throughput
- Understand the components of latency
- Understand cloud providers' physical infrastructure
- Solve problems at the correct layer

## What Backend Engineers Need to Know

### Abstraction Boundary

**Physical Layer vs Other Layers:**

| Layer | Responsibility | When Network Engineers Involved |
|------|------|-------------------|
| **Physical Layer** | Cables, fiber, signals | Link down, fiber cut |
| **Data Link Layer** | Ethernet, MAC, switching | VLAN, MTU |
| **Network Layer** | IP, routing, NAT | Subnet planning, routing policies |
| **Transport Layer** | TCP, UDP, ports | Connection management, performance tuning |
| **Application Layer** | HTTP, DNS, TLS | API design, performance optimization |

**When is Physical Layer Involved?**
- Data center selection
- CDN deployment
- Cross-region latency issues
- Network failures (fiber cut)

### Why is the Physical Layer Important?

**Scenario 1: Debugging High Latency**

```
API response slow: 200ms

Analysis:
- Application processing: 50ms
- Database query: 80ms
- Network latency: 70ms

Physical layer perspective:
- Transoceanic distance: 10,000 km
- Speed of light: 200,000 km/s (in fiber)
- Propagation latency: 50 ms (round trip)
- Remaining 20ms: Router processing, queuing

Conclusion:
- Physical limit: 50 ms (cannot optimize)
- Optimizable: Router processing, queuing (CDN, edge computing)
```

**Scenario 2: Selecting a Data Center**

```
User distribution:
- US: 40%
- Europe: 30%
- Asia Pacific: 30%

Single-region deployment (us-east-1):
- US users: 20ms
- Europe users: 150ms
- Asia Pacific users: 200ms
- Average latency: 97ms

Multi-region deployment:
- US (us-east-1): 20ms
- Europe (eu-west-1): 30ms
- Asia Pacific (ap-southeast-1): 40ms
- Average latency: 29ms

Conclusion:
- Physical distance determines latency lower bound
- Multi-region deployment can significantly improve user experience
```

## Bandwidth and Speed Concepts

### Bandwidth vs Throughput

**Bandwidth:** Theoretical maximum transmission rate

**Throughput:** Actual transmission rate

```
Bandwidth: 1 Gbps
  Overhead:
  - Ethernet header: 14 bytes
  - IP header: 20 bytes
  - TCP header: 20 bytes
  - Application data: 1460 bytes (MSS)
  Throughput: 1 Gbps  (1460 / 1514)  965 Mbps
```

**Why is 1 Gbps not equal to 1 Gbps of application data?**

```
Physical layer: 1 Gbps
  Ethernet overhead (2%)
Data link layer: 980 Mbps
  IP/UDP overhead (3%)
Network layer: 950 Mbps
  TCP overhead (3%)
Transport layer: 920 Mbps
  Application layer protocol (HTTP 2%)
Application layer: 900 Mbps
```

### Serialization Delay

**Definition:** Time to put data onto the physical link

**Calculation:**

```
Serialization delay = Packet size / Bandwidth

Example:
- 1500 byte packet, 1 Gbps link
- Serialization delay = 1500  8 bits / 1 Gbps = 12 s

- 1500 byte packet, 10 Mbps link
- Serialization delay = 1500  8 bits / 10 Mbps = 1.2 ms
```

**Impact:**
- Large packets on slow links have significant latency
- Small packets have minimal impact

## Latency Components

### Latency Breakdown

```
Total latency = Propagation delay + Processing delay + Queuing delay + Serialization delay
```

#### 1. Propagation Delay

**Definition:** Time for signal to propagate through medium

**Calculation:**

```
Propagation delay = Distance / Speed of light

Speed of light:
- Vacuum: 300,000 km/s
- Fiber: 200,000 km/s (refractive index 1.5)
- Copper: ~200,000 km/s

Example:
- Transoceanic distance: 10,000 km
- Fiber propagation delay: 10,000 / 200,000 = 50 ms (one-way)
- RTT: 100 ms
```

**Key Points:**
- Cannot optimize (physical limit)
- Longer distance, higher latency

#### 2. Processing Delay

**Definition:** Time for routers/switches to process packets

**Range:** Microseconds to milliseconds

**Optimization:**
- Faster hardware
- More efficient routing algorithms
- Hardware acceleration (ASIC, FPGA)

#### 3. Queuing Delay

**Definition:** Time packets wait in router queues

**Range:** Milliseconds to seconds (when congested)

**Optimization:**
- Increase bandwidth
- QoS (Quality of Service)
- Congestion control

**Scenario:**

```
No congestion: Queuing delay  0 ms
Light congestion: Queuing delay  10 ms
Moderate congestion: Queuing delay  100 ms
Severe congestion: Queuing delay > 1000 ms (packet loss)
```

#### 4. Serialization Delay

**Definition:** Time to put data onto the link

**Calculation:**

```
Serialization delay = Packet size / Bandwidth

Example:
- 1500 bytes, 1 Gbps: 12 s
- 1500 bytes, 10 Mbps: 1.2 ms
```

### Latency Example

**Scenario:** API request from New York to London

```
Distance: 5500 km

Propagation delay: 5500 / 200,000  2 (round trip) = 55 ms
Processing delay: Router processing (5 hops)  0.1 ms = 0.5 ms
Queuing delay: Light load  1 ms
Serialization delay: 1500 bytes / 1 Gbps  2 = 24 s

Total latency: 55 + 0.5 + 1 + 0.024  56.5 ms
```

**Analysis:**
- Propagation delay dominates (97%)
- Queuing delay may vary (when congested)
- Processing and serialization delays negligible

## Network Topology Impact on Backend Design

### Single Data Center vs Multi-Region

**Single Data Center:**

```
Users (global)

Data center (us-east-1)
  ├─ Database
  ├─ Cache
  └─ Application servers

Pros:
  - Simple architecture
  - Low cost
  - Strong data consistency

Cons:
  - High latency for distant users
  - Single point of failure

Suitable for:
  - Startups
  - Users concentrated in single region
```

**Multi-Region Deployment:**

```
US users  US region (us-east-1)
Europe users  Europe region (eu-west-1)
Asia Pacific users  Asia Pacific region (ap-southeast-1)

Inter-region synchronization:
  - Database master-slave replication
  - Cache synchronization
  - Message queue

Pros:
  - Low latency
  - High availability

Cons:
  - Complex architecture
  - Data consistency issues
  - High cost

Suitable for:
  - Global users
  - High availability requirements
```

### Edge Computing

**Purpose:** Push computation closer to users

```
Traditional architecture:
  Users  CDN (static resources)  Origin (dynamic content)

Edge computing:
  Users  Edge node (compute + cache)  Origin

Examples:
  - Cloudflare Workers
  - AWS Lambda@Edge
  - CloudFront Functions
```

**Advantages:**
- Lower latency
- Reduce origin load
- Better user experience

## Cloud Provider Physical Infrastructure

### AWS Regions and Availability Zones

**Region:**
- Geographically isolated areas
- Example: us-east-1 (Virginia), ap-southeast-1 (Singapore)

**Availability Zone (AZ):**
- Isolated data centers within a region
- Physical isolation (different buildings, different power)
- Low latency interconnection (< 1 ms)

```
us-east-1 (Virginia)
  ├─ us-east-1a (AZ A)
  ├─ us-east-1b (AZ B)
  └─ us-east-1c (AZ C)

Between AZs:
  - Physical isolation
  - Independent power
  - Low latency interconnection
```

### Cross-Region Latency

**Example: AWS Inter-Region Latency**

| Source Region | Destination Region | Latency |
|--------|---------|------|
| **us-east-1** | us-west-2 | 70 ms |
| **us-east-1** | eu-west-1 | 80 ms |
| **us-east-1** | ap-southeast-1 | 200 ms |
| **eu-west-1** | ap-southeast-1 | 180 ms |

**Design Impact:**

```
Single region:
  Read/write latency: < 1 ms

Cross-region master-slave:
  Write (master region): Local latency
  Read (slave region): 100 ms+

Cross-region active-active:
  Read/write latency: 100 ms+
  Data consistency: Eventual consistency
```

## Debugging Tools

### ping

**Measure latency:**

```bash
# Basic ping
ping -c 4 8.8.8.8

# Output:
# 64 bytes from 8.8.8.8: icmp_seq=1 ttl=118 time=15.2 ms
#                                ↑     ↑        ↑     ↑
#                                │     │        │     └─ RTT (round trip time)
#                                │     │        └─ TTL (time to live)
#                                │     └─ Sequence number
#                                └─ Packet size
```

### traceroute

**Discover routing path:**

```bash
traceroute google.com

# Output:
# 1  _gateway (192.168.1.1)  1.234 ms
# 2  10.0.0.1 (10.0.0.1)  15.234 ms
# 3  72.14.195.1 (72.14.195.1)  20.234 ms
# ...
```

### mtr

**Real-time path monitoring:**

```bash
mtr google.com

# Output:
# Host              Loss%   Snt   Last   Avg  Best  Wrst StDev
# 1. _gateway        0.0%    10    1.2   1.3   1.1   1.5   0.1
# 2. 10.0.0.1        0.0%    10   15.2  15.3  15.1  15.5   0.1
# 3. 72.14.195.1     0.0%    10   20.2  20.3  20.1  20.5   0.1
```

## Business Scenarios

### Scenario 1: Global User Latency Optimization

**Background:** Global users, single-region deployment in us-east-1

**Problem:**
- US users: 20ms
- Europe users: 150ms
- Asia Pacific users: 250ms

**Solutions:**

1. **CDN (Content Delivery Network)**
   - Static resources: Edge caching
   - Latency: 20-50ms

2. **Multi-Region Deployment**
   - Independent deployment per region
   - Latency: 20-50ms
   - Complexity: Data synchronization, consistency

3. **Edge Computing**
   - Dynamic content executed at edge
   - Latency: 20-50ms
   - Limitation: Limited compute capacity

---

### Scenario 2: Cross-Region Database Replication

**Background:** Master region us-east-1, slave region eu-west-1

**Problem:**
- After writing to master region, slave region reads old data
- Replication latency: 100ms

**Solutions:**

1. **Synchronous Replication**
   - Write waits for all regions to acknowledge
   - Latency: 100ms+
   - Availability: Reduced (any region failure, write fails)

2. **Asynchronous Replication**
   - Write returns immediately
   - Replication latency: 100ms
   - Consistency: Eventual consistency

3. **Application Layer Routing**
   - Write operations: Route to master region
   - Read operations: Route to nearest region
   - May read stale data
